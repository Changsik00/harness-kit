Plan Accept 전에 호출하여 spec.md의 품질을 독립 관점에서 검증합니다. 결과는 `critique.md`로 저장됩니다.

## 1. 현재 Spec 확인

```bash
bash scripts/harness/bin/sdd status --json
```

`spec` 필드로 spec 디렉토리를 특정합니다. active spec이 없으면 사용자에게 알리고 멈춥니다.

`specs/<spec-dir>/spec.md`가 존재하는지 확인합니다. 없으면 멈춥니다.

## 2. Opus 서브에이전트로 비판 수행

Agent tool (`subagent_type: general-purpose`, `model: "opus"`)을 사용하여 **별도 컨텍스트**에서 비판을 수행합니다.

서브에이전트에게 전달할 프롬프트:

> 당신은 독립적인 시니어 소프트웨어 아키텍트입니다. 주어진 spec.md를 비판적으로 검토하고 개선안을 제시하세요.
>
> **입력 자료**:
> 1. spec 문서: `specs/<spec-dir>/spec.md`를 읽어서 요구사항을 파악하세요.
> 2. 프로젝트 컨텍스트: `agent/constitution.md`와 `agent/agent.md`를 읽어 거버넌스를 파악하세요.
>
> 다음 3가지 관점에서 검토하세요:
>
> ### 1. 유사 기법 조사
> 웹 검색으로 이 spec이 해결하려는 문제와 동일/유사한 문제를 해결한:
> - 알려진 디자인 패턴 또는 아키텍처 패턴
> - 오픈소스 라이브러리 또는 도구
> - 업계 모범 사례(best practice)
>
> 검색 결과와 현재 spec의 접근법을 비교하세요.
>
> ### 2. 요구사항 비판
> - **누락**: 명시되지 않았지만 필요한 요구사항이 있는가?
> - **모순**: 서로 충돌하는 요구사항이 있는가?
> - **과잉 설계(YAGNI)**: 현재 문제 범위를 벗어난 요구사항이 있는가?
> - **모호함**: 구현 시 해석이 달라질 수 있는 표현이 있는가?
>
> ### 3. 대안 제안
> 현재 접근법과 다른 방향 2~3개를 제시하세요. 각 대안에 대해:
> - 핵심 아이디어 (1~2문장)
> - 장점
> - 단점
>
> 마지막에 **권장안**을 명시하세요 (현재 spec 유지 포함 가능).
>
> 출력 형식 (한국어):
> ```
> # Spec Critique: <spec-id>
>
> ## 1. 유사 기법 조사
> ### 발견된 패턴/도구
> - **<이름>**: <설명> — 현재 spec과의 비교: <차이점>
>
> ### 시사점
> <현재 접근법에 반영할 수 있는 인사이트>
>
> ## 2. 요구사항 비판
> ### 누락
> - <항목>: <이유>
>
> ### 모순
> - <항목>: <이유>
>
> ### 과잉 설계
> - <항목>: <이유>
>
> ### 모호함
> - <항목>: <이유>
>
> ## 3. 대안 제안
> ### 대안 A: <이름>
> - **아이디어**: ...
> - **장점**: ...
> - **단점**: ...
>
> ### 대안 B: <이름>
> ...
>
> ## 권장안
> <현재 spec 유지 / 대안 A / 대안 B 중 선택 + 이유>
> ```
>
> 결과는 반드시 **한국어**로 작성하세요.
> 발견된 것이 없는 항목은 "해당 없음"으로 표기하세요.

## 3. 결과 저장

비판 결과를 `specs/<spec-dir>/critique.md`에 저장합니다.

## 4. 사용자에게 보고

```
✅ Spec Critique 완료: <spec-id>
- 결과: specs/<spec-dir>/critique.md
- 주요 발견: <누락 N건 / 모순 N건 / 과잉 N건>
- 권장안: <현재 유지 / 대안 X>

Plan Accept 전에 critique.md를 검토하여 spec.md를 개선할 수 있습니다.
```
