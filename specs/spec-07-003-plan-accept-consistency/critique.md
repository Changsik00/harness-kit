# Spec Critique: spec-07-003

## 1. 유사 기법 조사

### 발견된 패턴/도구

- **Canonical Prompt Pattern (LLM Prompt Engineering)**: LLM 에이전트에게 특정 형식의 출력을 강제하기 위해 프롬프트에 정확한 템플릿을 삽입하는 기법. 현재 spec과의 비교: spec-07-003도 본질적으로 이 패턴을 따르고 있으며, `agent.md`에 고정 문구를 삽입하여 에이전트 출력을 표준화하는 접근법이다. 차이점은 없으며 정확히 이 패턴의 적용이다.

- **Slash Command Convention (Discord, Slack, GitHub)**: 사용자 입력을 `/command` 형식으로 표준화하여 기계적으로 파싱 가능하게 만드는 패턴. 현재 spec과의 비교: 이미 `/hk-plan-accept`라는 슬래시 커맨드가 존재하지만, 자연어 응답("ok", "yes")도 허용하려는 점이 다르다. 슬래시 커맨드 생태계에서는 보통 자연어 매핑 없이 커맨드만 인식한다.

- **Intent Recognition / NLU Fallback Pattern**: 챗봇 설계에서 특정 의도(intent)에 대해 여러 동의어/표현을 매핑하는 패턴. 사용자 입력의 다양성을 허용하면서도 의도를 정확히 파악한다. 현재 spec과의 비교: spec-07-003이 "허용 응답 목록"으로 하는 것이 바로 수동 intent mapping이다. 다만 실제 NLU 시스템은 fuzzy matching을 하지만, 이 spec은 정적 목록에 의존한다.

- **Confirmation Dialog Pattern (UI/UX)**: 위험한 작업 전에 사용자에게 명확한 확인을 요구하는 패턴. "Are you sure? [Yes/No]" 형식. 현재 spec과의 비교: Plan Accept가 바로 이 패턴의 CLI 버전이다. UI에서는 버튼이 명확하지만 텍스트 기반에서는 허용 입력을 명시해야 한다는 점에서 이 spec의 접근이 적절하다.

### 시사점

LLM 에이전트 컨텍스트에서 "허용 응답 목록"을 문서에 나열하는 것은 사실상 **프롬프트 엔지니어링**이다. 에이전트가 이 목록을 읽고 준수할지는 LLM의 지시 따르기(instruction following) 능력에 의존한다. 진짜 "인식 보장"이 필요하다면 프로그래매틱 검증(hook/스크립트)이 필요하지만, 이는 이 spec의 Out of Scope에 해당하므로 현 단계에서는 문서 기반 접근이 합리적이다.

## 2. 요구사항 비판

### 누락

- **목록 외 응답 수신 시 행동 규칙**: 허용 응답 목록은 정의했지만, 목록에 없는 응답(예: "진행해", "go", "ㅇㅇ")을 받았을 때 에이전트가 어떻게 해야 하는지 명시되지 않았다. "다시 물어보라"인지 "가장 유사한 의도로 해석하라"인지 정해야 한다.

- **spec-07-002와의 정합성 검증 기준**: 배경에서 "spec-07-002에서 추가한 선택 프롬프트와의 정합성도 필요"라고 언급했지만, Definition of Done에 이를 확인하는 체크리스트가 없다.

### 모순

- 해당 없음

### 과잉 설계

- **허용 목록 3곳 중복 (`agent.md`, `constitution.md`, `hk-plan-accept.md`)**: 동일한 목록을 3개 파일에 각각 기재하면 향후 변경 시 3곳을 모두 수정해야 한다. DRY 원칙 위배. 하나의 SSOT(Single Source of Truth)를 정하고 나머지는 참조하는 것이 바람직하다.

### 모호함

- **"대소문자 구분 없이 인식"의 범위**: LLM 지시인지 프로그래매틱 검증인지 모호하다. NFR2에서 "문서 변경만으로 구현"이라고 했으므로 결국 에이전트 프롬프트 준수에 의존하는 것인데, "인식"이라는 단어가 기계적 파싱을 암시할 수 있다.

- **`plan accept` 경계 조건**: `planaccept`(공백 없음), `Plan Accept`(혼합) 등 경계 조건이 불명확하다. LLM 기반이라 엄밀한 파싱은 아니지만, 문서에 나열하는 이상 범위는 명확해야 한다.

## 3. 대안 제안

### 대안 A: Single Source of Truth + 참조 방식

- **아이디어**: 허용 응답 목록을 `constitution.md` 한 곳에만 정의하고, `agent.md`와 `hk-plan-accept.md`는 "허용 응답은 constitution §4.2 참조"로 포인터만 둔다.
- **장점**: 유지보수 비용 최소화. 목록 변경 시 한 곳만 수정. 일관성 보장.
- **단점**: 에이전트가 constitution.md를 먼저 읽지 않으면 허용 응답을 모를 수 있다. 다만 bootstrap protocol에서 constitution.md를 반드시 읽으므로 실제 문제가 되지는 않는다.

### 대안 B: 프롬프트 간소화 (허용 목록 생략)

- **아이디어**: 목록을 나열하지 않고 "사용자가 긍정적 의사를 표현하면 Plan Accept로 처리한다"는 원칙 규칙만 둔다.
- **장점**: 문서가 간결해진다. 미래 표현에도 자연스럽게 대응. LLM의 자연어 이해 능력 활용.
- **단점**: "ok"가 Plan Accept인지 단순 acknowledgment인지 모호해질 수 있다. 현재 문제(에이전트가 인식하지 못해 다시 물어보는 현상)를 해결하지 못할 가능성이 있다.

### 대안 C: 번호 선택 전용 (자연어 배제)

- **아이디어**: `1` 또는 `2` 번호로만 응답받도록 강제. 번호 외 입력 시 다시 요청.
- **장점**: 완전히 명확한 인터페이스. 오인식 가능성 0.
- **단점**: 기존 사용자 습관("ok", "Y")을 깨뜨린다. NFR1("기존 워크플로우 마찰 없음")에 정면 위배.

## 권장안

**대안 A(SSOT + 참조 방식)를 적용한 현재 spec 개선**을 권장한다.

1. 현재 spec의 핵심 방향(표준 문구 + 허용 응답 목록)은 적절하다.
2. 다만 동일 목록을 3곳에 중복하는 것은 향후 불일치의 씨앗이므로, `constitution.md`를 SSOT로 삼고 나머지는 참조만 하도록 수정해야 한다.
3. 허용 목록 외 응답에 대한 행동 규칙("목록에 없는 응답 시 선택을 다시 요청한다")을 한 줄 추가할 것을 권고한다.
